# Claude AI 게임 생성 루틴

사용자가 "새 게임 추천해줘" 또는 유사한 요청을 하면 Claude는 이 절차를 따른다.

---

## 사용자 설정값 (확정)

| 항목 | 값 |
|---|---|
| 추천 게임 개수 | **3개** |
| 검증 실패 시 자동 수정 | **최대 3회** |
| 게임 데이터 출처 | **Claude 자동 생성** (사용자 승인 필요) |
| 커밋/푸시 타이밍 | **검증 통과 후 사용자 최종 확인** |
| 교과 연계 | **자유** (재미/교육성 위주) |

---

## 6단계 워크플로우

### 1단계: 현황 분석
```
입력: games/registry.json + index.html(CATEGORY_MAP)
출력: 분석 보고서
```

체크 항목:
- 카테고리별 게임 수 (speed/brain/math/knowledge/coop)
- 학년별 커버리지
- 메커니즘 중복도
- 부족한 영역 (예: "영어 게임 0개", "협력 게임 부족")

### 2단계: 신규 게임 추천 (3개)
분석 결과를 바탕으로 3개 후보를 제시. 각 후보는 다음 형식:

```
추천 N: {게임 이름}
- 카테고리: {카테고리}
- 패턴: {A | B | C}
- 메커니즘: {1줄 설명}
- 골든 템플릿: {flag-quiz | whack-a-mole | nim-game | secret-code}
- 부족 영역 해결: {예: "영어 교과 부재 해결"}
- 예상 제작 시간: {분 단위}
```

추천 우선순위:
1. 부족한 카테고리 우선
2. 새로운 메커니즘 (기존과 중복 회피)
3. 빠른 구현 가능성

### 3단계: 사용자 승인
- 사용자가 1개 이상 선택 ("1번 만들어줘")
- 또는 "다른 추천 더 보여줘" → 1단계로 복귀
- 승인된 게임에 대해 다음 단계 진행

### 4단계: 명세서 작성 + 데이터 생성
1. **명세서 작성** (`docs/GAME_SPEC.md` 양식 따라)
2. **콘텐츠 데이터 자동 생성**
   - data_count 만큼 문제/요소 생성
   - 사용자에게 샘플 5개 제시 → 승인 대기
   - 승인 시 → 5단계
   - 거부 시 → 수정 후 재제시

### 5단계: 게임 제작
1. 골든 템플릿 폴더 복사 (`games/{template}/` → `games/{folder}/`)
2. 파일별 수정:
   - `game.json` — 메타데이터 교체
   - `index.html` — 제목/일러스트/문구 교체
   - `style.css` — 게임 고유 스타일 (대부분 그대로)
   - `game.js` — 데이터 + 정답 판정 로직 교체
3. 등록:
   - `games/registry.json`에 폴더명 추가
   - `index.html`(런처)의 CATEGORY_MAP, GAME_ICONS, FALLBACK_GAMES 추가

### 6단계: 검증
`scripts/verify-game.js`를 실행해 자동 검증. 검증 항목:

**정적 검증** (스크립트):
- [ ] 4개 파일 존재 (game.json, index.html, style.css, game.js)
- [ ] game.json 필수 필드 존재 (name, description, icon, color, grades, playTime)
- [ ] index.html에 4개 screen 존재 (intro, countdown, game, result)
- [ ] index.html에 shared/style.css, shared/engine.js 링크
- [ ] registry.json에 폴더명 등록됨
- [ ] launcher index.html의 CATEGORY_MAP, FALLBACK_GAMES에 등록됨
- [ ] JS 문법 오류 없음 (node --check)

**브라우저 검증** (Preview MCP):
- [ ] 메인 페이지에서 새 게임 카드 표시
- [ ] 카드 클릭 → 인트로 화면 정상 로드
- [ ] PLAY 버튼 → 카운트다운 → 게임 화면 전환
- [ ] 2P/3P/4P 모든 인원 모드 동작
- [ ] X 닫기 버튼 위치 정상 (zone과 겹침 없음)
- [ ] 점수 계산 정확
- [ ] 결과 화면 정상 표시
- [ ] 콘솔 에러 0개
- [ ] 다시하기/홈으로 버튼 동작

**검증 실패 시:**
- 최대 3회 자동 수정 시도
- 시도 1: 명백한 버그 (오타, 누락) 수정
- 시도 2: 로직 수정 (정답 판정, 화면 전환)
- 시도 3: 마지막 시도
- 3회 실패 시 → 사용자에게 보고 + 수동 개입 요청

### 7단계: 사용자 최종 확인 + 배포
1. 검증 통과 보고 + 스크린샷
2. 사용자: "푸시해줘" → git commit + push
3. 사용자: "수정해줘" → 추가 수정 후 재검증

## Claude의 행동 규칙

### 🚫 절대 하지 말 것
- 사용자 승인 없이 게임 제작 시작
- 데이터 샘플 미리 보여주지 않고 적용
- 검증 실패를 숨기고 통과로 보고
- 자동 푸시/배포
- **새로운 디자인 도입** (색상, 폰트, 버튼 모양, 레이아웃 등)
- **`shared/style.css` 수정** (전체 게임에 영향 — 매우 위험)
- **HTML 구조 변경** (3-screen 골격 깨기)
- **CSS에 임의 색상/그림자/폰트 추가**
- **`!important` 남용해서 공통 스타일 덮어쓰기**

### ✅ 반드시 할 것
- 매 단계마다 사용자에게 진행 상황 보고
- 3회 자동 수정 실패 시 즉시 멈춤
- 콘솔 에러 발견 시 즉시 보고

## 🎨 디자인 일관성 규칙 (최상위 원칙)

> **새 게임은 기존 게임과 디자인이 100% 동일해야 한다. 게임 콘텐츠만 다르다.**

### 강제 사항

1. **반드시 골든 템플릿을 복사로 시작**
   - 처음부터 새로 작성 금지
   - 패턴 A → `flag-quiz` 폴더 통째로 복사
   - 패턴 B → `whack-a-mole` 폴더 통째로 복사
   - 패턴 C → `nim-game` 또는 `secret-code` 폴더 통째로 복사

2. **HTML 구조는 그대로 유지**
   - `<section class="screen screen-intro">` 등 4개 screen 클래스명 동일
   - `.game-header`, `.intro-body`, `.btn-play`, `.zones-wrap`, `.score-bar` 등 공통 클래스 그대로
   - 기존 클래스명을 임의로 바꾸지 않음

3. **CSS는 "교체"가 아니라 "최소 수정"**
   - 게임 고유 요소(예: 국기 이미지 표시 영역)만 추가
   - 색상은 반드시 디자인 토큰 사용 (CSS 변수 또는 기존 파스텔)
   - **금지된 변경**:
     - 배경색을 새 색으로 변경
     - 테두리 두께 변경 (3px 고정)
     - font-weight 변경 (900 고정)
     - 그림자 스타일 변경 (오프셋 그림자 고정)
     - 둥근 모서리 반지름 임의 변경
     - 새 폰트 도입

4. **JS 로직만 자유롭게 변경**
   - 데이터 (문제/정답)
   - 정답 판정 로직
   - 라운드 진행 방식
   - 점수 계산
   - SVG 일러스트 (인트로 화면)
   - **단, 화면 전환 함수와 공통 모듈 호출 패턴은 유지**

5. **검증 단계에서 디자인 일관성 체크**
   - 기존 게임 스크린샷과 새 게임 스크린샷을 같은 인원수(2P)에서 비교
   - 다음이 동일한지 확인:
     - [ ] 인트로 화면 헤더 모양
     - [ ] PLAY 버튼 색/크기/위치
     - [ ] 게임 화면 X 닫기 버튼 위치
     - [ ] zone 배경색 (파스텔 4색만 사용)
     - [ ] 결과 화면 다시하기/홈 버튼 모양
     - [ ] 폰트, 그림자, 테두리 두께

### 디자인 일관성 위반 시 처리
- 검증 실패 → 즉시 골든 템플릿과 diff 확인 → 차이점 원복
- 사용자가 "디자인 다르다" 라고 지적 → 즉시 골든 템플릿 기준으로 재작성

### 허용되는 게임별 차이
- 인트로 일러스트 SVG (게임마다 다른 그림)
- 게임 영역 안의 콘텐츠 (국기/단어/숫자/카드 등)
- 게임 고유 인터랙션 요소 (예: nim-game의 take-buttons)
- 효과음 (게임 분위기에 맞게)

> 즉, **"껍데기는 똑같고 안에 들어가는 콘텐츠만 다른 형제 게임"**을 만드는 것이 목표.

# 🤖 완전 자동 모드 (Autonomous Mode)

수동 모드와 별개로, **완전 자동 모드**가 매일 한국시간 오전 10시에 실행된다.

## 사용자 설정값 (확정)

| 항목 | 값 |
|---|---|
| 자율성 | **완전 자동** (푸시까지 사용자 확인 없음) |
| 게임 선택 | Claude 자동 결정 (아래 알고리즘) |
| 한 번에 만들 게임 수 | **1개** |
| 종료 조건 | **무기한** (수동 중지 명령까지) |
| 연속 실패 차단 | **3개 연속 실패 시** 그날 작업 중단 |
| 중복 방지 | **느슨** (폴더명 중복만 회피) |
| 알림 | **GitHub 커밋 로그만** |
| 푸시 실패 처리 | **로컬 커밋 유지** + 다음 실행 재시도 |
| 자가 품질 게이트 | **엄격** (Claude 정의, 아래 6개 기준) |
| 트리거 | 매일 10:00 KST 자동 + 수동 명령 둘 다 |

## 자동 선택 알고리즘 (Claude가 정의)

### 우선순위 (위에서 아래로)
1. **부족 카테고리** — 게임 수 가장 적은 카테고리 우선
   - 현재 분포: speed(4) brain(8) math(6) knowledge(6) coop(2)
   - coop이 가장 부족 → coop 후보 우선 검토
2. **메커니즘 다양성** — 기존 게임과 가장 다른 메커니즘 선호
   - 같은 패턴(A/B/C) 안에서도 데이터 형식이 겹치면 회피
3. **빠른 구현 가능성** — 골든 템플릿(flag-quiz / whack-a-mole / nim-game / secret-code) 기반으로 만들 수 있는 게임 우선
4. **교육 가치** — 초등 교과 연계되면 가산점

### 게임 후보 풀 (자동 모드용 백로그)
Claude는 다음 후보 중에서 위 우선순위에 따라 선택:

**coop 후보 (현재 부족, 우선)**
- 박자 따라하기 (rhythm-echo) — 청각 협력
- 같은 그림 찾기 (find-pair-coop) — 정보 비대칭 협력
- 색깔 신호 보내기 (color-signal) — 시각 신호 전달

**knowledge 확장 후보**
- 동물 분류 (animal-sort) — 과학
- 한글 자음 모음 (hangul-jamo) — 국어
- 사칙연산 부호 (math-sign) — 수학 (knowledge 분류는 부적합 → math)

**brain 신선 후보**
- 패턴 잇기 (pattern-chain) — 시퀀스 추론
- 짝 맞추기 강화판 (memo-twin) — 기억력

**speed 신선 후보**
- 빠르게 분류 (quick-sort-revisit) — 좌/우 분류 반사

선택 시 동일 폴더명 사용 금지. 만약 모든 후보가 폴더명 중복이면 "후보 없음" 보고 후 종료.

## 자가 품질 게이트 (Claude 정의 6개 기준)

검증 통과해도 다음 중 하나라도 위반하면 게임 폐기:

1. **데이터 30개 미만** — 패턴 A는 보기 다양성 위해 30개 이상 필수
2. **보기 4개 중 정답 중복** — `currentChoices`에 정답이 2번 이상 들어가면 거부
3. **인트로 일러스트 누락** — `intro-body` 안에 SVG 또는 이미지 요소 없으면 거부
4. **shared/style.css 수정** — 공통 스타일 건드린 흔적 발견 시 거부 (위험)
5. **콘솔 에러** — 브라우저 검증 중 1개라도 발견 시 거부
6. **결과 화면 도달 실패** — 10라운드 자동 플레이 후 resultScreen 활성화 안 되면 거부

## 자동 모드 워크플로우 (8단계)

```
[1. 분석] → [2. 자동 선택] → [3. 명세+데이터] → [4. 제작] →
[5. 자가 품질 검증] → [6. 정적+브라우저 검증] → [7. 자동 커밋] → [8. 푸시 + 로그]
```

각 단계에서 실패하거나 게이트 통과 못하면:
- **게임 폐기** — 폴더 삭제 + registry/launcher 등록 해제
- **다음 후보 재시도** — 최대 3개까지
- **3개 연속 실패** — 그날 작업 중단, 다음날 10시 다시 시도

## 자동 모드 보고 형식

자동 실행 결과는 **git commit message**에 다음 형식으로 기록:
```
Auto-add: {게임 이름} ({카테고리}, 패턴 {A|B|C})

선택 근거: {부족 카테고리|메커니즘 다양성|...}
데이터: {n}개, {카테고리 분류}
검증: 정적 17/17, 브라우저 정상, 자가 게이트 6/6
실행: 자동 (KST {timestamp})

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```
