# Claude AI 게임 생성 루틴

사용자가 "새 게임 추천해줘" 또는 유사한 요청을 하면 Claude는 이 절차를 따른다.

---

## ⚠️ 절대 규칙 #0 — 디자인 통일 (Level 3 Comic 스타일)

**모든 게임은 카테고리·패턴 무관하게 다음 디자인을 따른다. 위반 시 즉시 거부.**

### 색상 팔레트 (이외 색상 사용 금지)
- **배경**: 크림(#FFF8E1, #FFF8F0) 또는 흰색 (#FFFFFF)
- **zone 배경**: 파스텔 4색만 — `#B3E5FC`(p1), `#FFCDD2`(p2), `#C8E6C9`(p3), `#FFE0B2`(p4)
- **테두리**: 검정 (#2C2C2C) 두께 3px
- **그림자**: 오프셋 하드 그림자 (예: `box-shadow: 6px 6px 0 #2C2C2C`)
- **포인트 색**: 노랑(#FFD54F), 보라(#7C4DFF), 초록(#4CAF50)
- **텍스트**: 검정 #2C2C2C, 부제 #555/#666

### 절대 금지 (퍼즐 게임 추가 시 특히 주의)
- ❌ **다크 배경** (#000, #0D1B2A, #1A1A1A 등) — 일반 퍼즐 앱 스타일
- ❌ **네온 색상** (#00FF00, #FF00FF, 형광색) — 게임센터 스타일
- ❌ **그라디언트 배경** (지나친 색 변화)
- ❌ **얇은 테두리** (1px, 2px) — 반드시 3px
- ❌ **부드러운 그림자** (`box-shadow: 0 2px 8px rgba(0,0,0,0.1)`) — 반드시 오프셋 하드
- ❌ **둥근 모서리 무한대** (`border-radius: 50%` 외 큰 값) — 카드는 14-22px
- ❌ **새 폰트 도입** — Pretendard Variable만 사용
- ❌ **font-weight 700 이하** — 굵은 글씨는 800-900
- ❌ **shared/style.css 수정** — 공통 스타일 절대 건드리지 말 것

### 퍼즐 카테고리 추가 시 특별 주의
일반적인 퍼즐 앱은 다크 테마 + 네온 색이 표준이지만, **이 프로젝트는 절대 그 톤을 따르지 않는다**:

| 보통의 퍼즐 앱 | 짬짬이 교실 (Level 3) |
|---|---|
| 검정 배경 + 네온 그리드 | 크림 배경 + 검정 격자선 |
| 글로우 이펙트 | 오프셋 하드 그림자 |
| 미니멀 얇은 선 | 두꺼운 검정 테두리 |
| 화려한 그라디언트 | 단색 파스텔 |
| 소프트 효과음 | 청키한 효과음 (기존과 동일) |

### 검증 시 강제 체크
새 게임 검증 단계에서 다음 항목을 추가 점검:
- [ ] CSS에 `#000`/`#1A1A1A`/`#0D1B2A` 등 다크 배경 색 없음
- [ ] CSS에 `gradient` 키워드 (인트로 헤더 외) 사용 없음
- [ ] CSS에 `border-radius: 50%` 외 30px 초과 없음
- [ ] CSS에 `box-shadow` 중 오프셋 0이거나 흐릿한 (`rgba(0,0,0,0.1)` 약한) 그림자 없음
- [ ] 모든 텍스트 `font-weight` 800 이상 (보통 텍스트 700 가능)

### 위반 시 처리
검증 단계에서 위 항목 중 **1개라도 발견**되면:
1. 즉시 폐기
2. 자동 모드: discard + 다음 후보
3. 수동 모드: 사용자에게 보고 + 수정안 제시

## ⚠️ 절대 규칙 #0-2 — 터치스크린 학생 접근성

**모든 게임은 태블릿/터치스크린을 둘러싼 초등학생이 자연스럽게 조작할 수 있어야 한다.**

### 핵심 원칙: 인터랙션 요소는 zone 하단에 배치

학생들은 태블릿을 책상에 놓고 둘러서서 게임을 한다. 손이 자연스럽게 닿는 곳은 **각 zone의 하단**이다. 화면 상단의 요소를 조작하려면 손을 뻗어야 하고, 4P 모드에서는 다른 학생과 부딪힐 수도 있다.

### 강제 규칙

1. **인터랙션 요소(보기 버튼, 퍼즐, 그리드, 드래그 영역)는 zone 하단에 위치**
   - 헤더(라벨, 점수칩) → 상단
   - 안내/문제 표시 → 상단 또는 중간
   - **터치 조작 영역 → 하단** (margin-top: auto 또는 justify-content: flex-end)

2. **상단 영역은 보기 전용**
   - 문제 텍스트, 타이머, 카운터, 카테고리 배지 등 정보 표시만
   - 터치 안 해도 되는 요소만 위에 배치

3. **zone 하단 패딩은 최소 12-16px**
   - 화면 가장자리에 너무 붙으면 터치 정확도 저하

4. **버튼/타일 최소 크기 44x44px**
   - 어린 학생 손가락 정확도 기준

### 패턴별 적용

| 패턴 | 상단 (정보) | 하단 (조작) |
|---|---|---|
| A 동시반응 | 문제 표시 | 보기 버튼 4개 |
| B 개별영역 | 점수, 타이머 | 게임 요소 (두더지/풍선 등) |
| C 협력 | 역할 표시 | 입력/선택 영역 |
| D 퍼즐 | 라운드/타이머 | **퍼즐 그리드/드래그 영역** |

### CSS 예시 (모든 새 게임에서 따를 것)

```css
.zone {
  display: flex;
  flex-direction: column;
  /* ... */
}

.zone-header { /* 자동으로 상단 */ }

.{interactive-element} {
  margin-top: auto;  /* zone 하단으로 밀어냄 */
}
```

또는:

```css
.zone {
  justify-content: space-between;  /* 상단/하단 분리 */
}
```

### 검증 시 강제 체크
- [ ] 게임 화면에서 인터랙션 요소(클릭 가능한 영역)의 Y 좌표가 zone 중앙 이하
- [ ] zone 하단 여백이 zone 상단 여백보다 작거나 비슷
- [ ] 4P 모드에서도 같은 원칙 유지

### 위반 사례
- ❌ 그리드/버튼이 zone 상단에 붙어 있고 아래 빈 공간이 큼
- ❌ 화면 정중앙(브라우저 기준)에 인터랙션 요소 배치 (큰 태블릿에서 손 안 닿음)
- ❌ 하단에 점수 바 + 그 위에 큰 빈 공간 + 그 위에 버튼

## 6단계 워크플로우

### 1단계: 현황 분석
```
입력: games/registry.json + index.html(CATEGORY_MAP)
출력: 분석 보고서
```

체크 항목:
- 카테고리별 게임 수 (speed/brain/math/knowledge/coop/puzzle)
- 학년별 커버리지
- 메커니즘 중복도
- 부족한 영역 (예: "영어 게임 0개", "협력 게임 부족")

### 2단계: 신규 게임 추천 (3개)
분석 결과를 바탕으로 3개 후보를 제시. 각 후보는 다음 형식:

```
추천 N: {게임 이름}
- 카테고리: {카테고리}  (speed/brain/math/knowledge/coop/puzzle)
- 패턴: {A | B | C | D}
- 메커니즘: {1줄 설명}
- 골든 템플릿: {flag-quiz | whack-a-mole | nim-game | secret-code | slide-puzzle}
- 부족 영역 해결: {예: "영어 교과 부재 해결"}
- 예상 제작 시간: {분 단위}
```

추천 우선순위:
1. 부족한 카테고리 우선
2. 새로운 메커니즘 (기존과 중복 회피)
3. 빠른 구현 가능성

### 3단계: 사용자 승인
- 사용자가 1개 이상 선택 ("1번 만들어줘")
- 또는 "다른 추천 더 보여줘" → 1단계로 복귀
- 승인된 게임에 대해 다음 단계 진행

### 4단계: 명세서 작성 + 데이터 생성
1. **명세서 작성** (`docs/GAME_SPEC.md` 양식 따라)
2. **콘텐츠 데이터 자동 생성**
   - data_count 만큼 문제/요소 생성
   - 사용자에게 샘플 5개 제시 → 승인 대기
   - 승인 시 → 5단계
   - 거부 시 → 수정 후 재제시

### 5단계: 게임 제작
1. 골든 템플릿 폴더 복사 (`games/{template}/` → `games/{folder}/`)
2. 파일별 수정:
   - `game.json` — 메타데이터 교체
   - `index.html` — 제목/일러스트/문구 교체
   - `style.css` — 게임 고유 스타일 (대부분 그대로)
   - `game.js` — 데이터 + 정답 판정 로직 교체
3. 등록:
   - `games/registry.json`에 폴더명 추가
   - `index.html`(런처)의 CATEGORY_MAP, GAME_ICONS, FALLBACK_GAMES 추가

### 6단계: 검증
`scripts/verify-game.js`를 실행해 자동 검증. 검증 항목:

**정적 검증** (스크립트):
- [ ] 4개 파일 존재 (game.json, index.html, style.css, game.js)
- [ ] game.json 필수 필드 존재 (name, description, icon, color, grades, playTime)
- [ ] index.html에 4개 screen 존재 (intro, countdown, game, result)
- [ ] index.html에 shared/style.css, shared/engine.js 링크
- [ ] registry.json에 폴더명 등록됨
- [ ] launcher index.html의 CATEGORY_MAP, FALLBACK_GAMES에 등록됨
- [ ] JS 문법 오류 없음 (node --check)

**브라우저 검증** (Preview MCP):
- [ ] 메인 페이지에서 새 게임 카드 표시
- [ ] 카드 클릭 → 인트로 화면 정상 로드
- [ ] PLAY 버튼 → 카운트다운 → 게임 화면 전환
- [ ] 2P/3P/4P 모든 인원 모드 동작
- [ ] X 닫기 버튼 위치 정상 (zone과 겹침 없음)
- [ ] 점수 계산 정확
- [ ] 결과 화면 정상 표시
- [ ] 콘솔 에러 0개
- [ ] 다시하기/홈으로 버튼 동작

**검증 실패 시:**
- 최대 3회 자동 수정 시도
- 시도 1: 명백한 버그 (오타, 누락) 수정
- 시도 2: 로직 수정 (정답 판정, 화면 전환)
- 시도 3: 마지막 시도
- 3회 실패 시 → 사용자에게 보고 + 수동 개입 요청

### 7단계: 사용자 최종 확인 + 배포
1. 검증 통과 보고 + 스크린샷
2. 사용자: "푸시해줘" → git commit + push
3. 사용자: "수정해줘" → 추가 수정 후 재검증

## Claude의 행동 규칙

### 🚫 절대 하지 말 것
- 사용자 승인 없이 게임 제작 시작
- 데이터 샘플 미리 보여주지 않고 적용
- 검증 실패를 숨기고 통과로 보고
- 자동 푸시/배포
- **새로운 디자인 도입** (색상, 폰트, 버튼 모양, 레이아웃 등)
- **`shared/style.css` 수정** (전체 게임에 영향 — 매우 위험)
- **HTML 구조 변경** (3-screen 골격 깨기)
- **CSS에 임의 색상/그림자/폰트 추가**
- **`!important` 남용해서 공통 스타일 덮어쓰기**

### ✅ 반드시 할 것
- 매 단계마다 사용자에게 진행 상황 보고
- 3회 자동 수정 실패 시 즉시 멈춤
- 콘솔 에러 발견 시 즉시 보고

## 🎨 디자인 일관성 규칙 (최상위 원칙)

> **새 게임은 기존 게임과 디자인이 100% 동일해야 한다. 게임 콘텐츠만 다르다.**

### 강제 사항

1. **반드시 골든 템플릿을 복사로 시작**
   - 처음부터 새로 작성 금지
   - 패턴 A → `flag-quiz` 폴더 통째로 복사
   - 패턴 B → `whack-a-mole` 폴더 통째로 복사
   - 패턴 C → `nim-game` 또는 `secret-code` 폴더 통째로 복사

2. **HTML 구조는 그대로 유지**
   - `<section class="screen screen-intro">` 등 4개 screen 클래스명 동일
   - `.game-header`, `.intro-body`, `.btn-play`, `.zones-wrap`, `.score-bar` 등 공통 클래스 그대로
   - 기존 클래스명을 임의로 바꾸지 않음

3. **CSS는 "교체"가 아니라 "최소 수정"**
   - 게임 고유 요소(예: 국기 이미지 표시 영역)만 추가
   - 색상은 반드시 디자인 토큰 사용 (CSS 변수 또는 기존 파스텔)
   - **금지된 변경**:
     - 배경색을 새 색으로 변경
     - 테두리 두께 변경 (3px 고정)
     - font-weight 변경 (900 고정)
     - 그림자 스타일 변경 (오프셋 그림자 고정)
     - 둥근 모서리 반지름 임의 변경
     - 새 폰트 도입

4. **JS 로직만 자유롭게 변경**
   - 데이터 (문제/정답)
   - 정답 판정 로직
   - 라운드 진행 방식
   - 점수 계산
   - SVG 일러스트 (인트로 화면)
   - **단, 화면 전환 함수와 공통 모듈 호출 패턴은 유지**

5. **검증 단계에서 디자인 일관성 체크**
   - 기존 게임 스크린샷과 새 게임 스크린샷을 같은 인원수(2P)에서 비교
   - 다음이 동일한지 확인:
     - [ ] 인트로 화면 헤더 모양
     - [ ] PLAY 버튼 색/크기/위치
     - [ ] 게임 화면 X 닫기 버튼 위치
     - [ ] zone 배경색 (파스텔 4색만 사용)
     - [ ] 결과 화면 다시하기/홈 버튼 모양
     - [ ] 폰트, 그림자, 테두리 두께

### 디자인 일관성 위반 시 처리
- 검증 실패 → 즉시 골든 템플릿과 diff 확인 → 차이점 원복
- 사용자가 "디자인 다르다" 라고 지적 → 즉시 골든 템플릿 기준으로 재작성

### 허용되는 게임별 차이
- 인트로 일러스트 SVG (게임마다 다른 그림)
- 게임 영역 안의 콘텐츠 (국기/단어/숫자/카드 등)
- 게임 고유 인터랙션 요소 (예: nim-game의 take-buttons)
- 효과음 (게임 분위기에 맞게)

> 즉, **"껍데기는 똑같고 안에 들어가는 콘텐츠만 다른 형제 게임"**을 만드는 것이 목표.

# 🤖 완전 자동 모드 (Autonomous Mode)

수동 모드와 별개로, **완전 자동 모드**가 매일 한국시간 오전 10시에 실행된다.

## 사용자 설정값 (확정)

| 항목 | 값 |
|---|---|
| 자율성 | **완전 자동** (푸시까지 사용자 확인 없음) |
| 게임 선택 | Claude 자동 결정 (아래 알고리즘) |

| 트리거 | 매일 10:00 KST 자동 + 수동 명령 둘 다 |

## 자동 선택 알고리즘 (Claude가 정의)

### 우선순위 (위에서 아래로)
1. **부족 카테고리** — 게임 수 가장 적은 카테고리 우선
   - 현재 분포: speed(4) brain(8) math(6) knowledge(6) coop(3) puzzle(6)
   - 매번 분석 시점에 `node scripts/auto-add-game-helpers.js stats`로 재확인
2. **메커니즘 다양성** — 기존 게임과 가장 다른 메커니즘 선호
   - 같은 패턴(A/B/C/D) 안에서도 데이터 형식이 겹치면 회피
3. **빠른 구현 가능성** — 골든 템플릿(flag-quiz / whack-a-mole / nim-game / secret-code / slide-puzzle) 기반으로 만들 수 있는 게임 우선
4. **교육 가치** — 초등 교과 연계되면 가산점

### 게임 후보 풀 (자동 모드용 백로그) — 38개

> **이 풀에 적힌 게임만 자동 제작된다.** (블랙박스 1회용 자율 생성 폐지)
> 자동 모드는 아래 후보 중 우선순위에 따라 선택하며, 이미 `registry.json`에 구현된 동명 게임은 자동 제외(`list-existing-folders`로 확인).
> **풀이 모두 소진되면** → 아래 "풀 재생성 규칙"에 따라 새 40개를 발상해 이 섹션을 통째로 교체한다.

**speed 후보 (6)**
- 연타 대결 (tap-rush, A) — 제한 5초간 버튼을 최대한 많이 연타, 횟수 경쟁
- 빛 따라잡기 (light-chase, A) — 원을 도는 빛이 목표 칸을 지나는 순간 정확히 터치
- 빨간불 멈춤 (traffic-stop, A) — 초록불에 터치, 빨간불엔 손 떼고 기다리기 (자제력)
- 공 받기 (falling-catch, B) — 좌우로 움직여 떨어지는 공 받고 폭탄은 피하기
- 장애물 피하기 (dodge-run, B) — 달리며 다가오는 장애물을 점프·숙이기로 회피
- 색 맞춰 터치 (color-switch, A) — 화면 배경색과 같은 색 버튼을 빠르게 (순수 색매칭+속도)

**brain 후보 (7)**
- 다음 무늬 (pattern-next, A) — 반복되는 도형 패턴의 다음에 올 것 고르기
- 블록 세기 (block-count, A) — 쌓여서 일부 가려진 블록의 전체 개수 추론
- 불 끄기 (light-out, D) — 누르면 이웃 칸도 토글, 모든 불 끄기
- 미니 스도쿠 (sudoku-mini, D) — 4×4 스도쿠
- 다른 하나 (odd-one-out, A) — 크기·방향·종류가 다른 하나 찾기 (색만 다른 odd-color와 구별)
- 컵 섞기 (cup-shuffle, A) — 공 숨긴 컵을 섞은 뒤 위치 맞히기 (집중·추적)
- 짝 찾기 (relation-match, A) — 엄마-새끼, 도구-직업 등 관계 연결

**math 후보 (7)**
- 수 가르기 (number-bond, A) — 10을 두 수로 가르기·모으기 빈칸 채우기
- 뛰어 세기 (skip-count, A) — 2씩·5씩 뛰어 센 수열의 빈칸
- 동전 세기 (coin-count, A) — 동전 그림의 합계 계산
- 분수 피자 (fraction-pizza, A) — 색칠된 조각으로 분수 인식
- 주사위 합 (dice-sum, A) — 두 주사위 눈 더하기
- 길이 재기 (measure-length, A) — 눈금자로 길이 읽고 비교
- 십틀 채우기 (ten-frame, A) — 십틀의 점 개수로 수 세기

**knowledge 후보 (7)**
- 비슷한 말 (synonym-quiz, A) — 유의어 고르기 (반대말 opposite-word의 짝)
- 직업과 도구 (job-tool, A) — 도구 보고 직업 맞히기
- 계절 맞히기 (season-quiz, A) — 풍경·활동으로 계절 추론
- 몸의 이름 (body-name, A) — 신체 부위 명칭
- 단위 세기 (unit-noun, A) — 한 마리·한 권 등 단위명사 (국어)
- 사자성어 (idiom-quiz, A) — 사자성어 뜻 맞히기
- 음식 재료 (food-origin, A) — 음식의 주재료 맞히기

**coop 후보 (5)**
- 미로 안내 (maze-guide, C) — P1만 전체 지도, 말로 P2를 출구까지 안내
- 탑 쌓기 (tower-stack, C) — 번갈아 블록을 올려 무너지지 않게 균형 유지
- 기억 이어가기 (memory-relay, C) — 번갈아 색 시퀀스에 1개씩 추가하며 따라하기 (시각·누적)
- 반쪽 맞추기 (half-match, C) — P1 왼쪽 반·P2 오른쪽 반 골라 한 그림 완성
- 길 이어 그리기 (path-relay, C) — P1이 경로 절반, P2가 이어서 목표까지

**puzzle 후보 (6)**
- 칠교놀이 (tangram, D) — 7조각으로 실루엣 채우기
- 블록 채우기 (block-fit, D) — 테트로미노 조각으로 빈칸 메우기
- 상자 밀기 (sokoban, D) — 상자를 목표 위치로 밀어 넣기
- 하노이탑 (hanoi, D) — 원반을 규칙대로 옮겨 탑 이동
- 차 빼기 (unblock, D) — 막힌 차들을 밀어 출구로 빼내기
- 종이접기 (fold-guess, A) — 접은 종이를 펼치면 나올 모양 추측

#### 풀 재생성 규칙 (40개 소진 시)
위 40개가 모두 `registry.json`에 구현되어 풀이 비면, 그날 **블랙박스로 1개를 짜내지 말고** 다음을 수행해 풀을 다시 40개로 채운다:
1. `stats`로 카테고리 분포 확인
2. **대조 대상 2개를 모두 확보**: ⓐ `registry.json`의 현재 전체 게임, ⓑ 위 "🚫 사용자 삭제" 목록
3. ⓐ·ⓑ와 **폴더명도 메커니즘도 겹치지 않는** 새 40개를 카테고리 균형 있게(각 6~7개) 발상
4. 이 "게임 후보 풀" 섹션을 통째로 교체하고 커밋 (`docs: 후보 풀 재생성 (40개)`)
5. 그 후 평소 절차대로 풀에서 골라 제작
> 핵심: 만들어지는 모든 게임은 **항상 이 풀에 먼저 기록**되므로 사용자가 사전 검토할 수 있고, 삭제 목록과 대조되어 재추가가 차단된다.

## 자가 품질 게이트 (Claude 정의 6개 기준)

검증 통과해도 다음 중 하나라도 위반하면 게임 폐기:

1. **데이터 30개 미만** — 패턴 A는 보기 다양성 위해 30개 이상 필수
2. **보기 4개 중 정답 중복** — `currentChoices`에 정답이 2번 이상 들어가면 거부
3. **인트로 일러스트 누락** — `intro-body` 안에 SVG 또는 이미지 요소 없으면 거부
4. **shared/style.css 수정** — 공통 스타일 건드린 흔적 발견 시 거부 (위험)
5. **콘솔 에러** — 브라우저 검증 중 1개라도 발견 시 거부
6. **결과 화면 도달 실패** — 10라운드 자동 플레이 후 resultScreen 활성화 안 되면 거부

## 자동 모드 워크플로우 (8단계)

```
[1. 분석] → [2. 자동 선택] → [3. 명세+데이터] → [4. 제작] →
[5. 자가 품질 검증] → [6. 정적+브라우저 검증] → [7. 자동 커밋] → [8. 푸시 + 로그]
```

각 단계에서 실패하거나 게이트 통과 못하면:
- **게임 폐기** — 폴더 삭제 + registry/launcher 등록 해제
- **다음 후보 재시도** — 최대 3개까지
- **3개 연속 실패** — 그날 작업 중단, 다음날 10시 다시 시도
